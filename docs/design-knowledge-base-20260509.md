## Problem Statement

构建一个个人科研工作流知识库，具备以下核心能力：
- 文件导入：PDF、DOCX、Markdown、文件夹（含 claude-mem 导出的 .db/.wal 数据库）
- 本地优先的 AI 推理（llama.cpp 集成），云端作为可选降级
- 自定义知识板块（不限数量，用户按技能/知识领域组织）
- 板块专属 system prompt（约束不同板块下 AI 的输出格式和风格）
- Chat 对话界面，支持本地/云端模型切换
- Chat 自动摘要（5000字基准，阶梯扩展）
- 画像系统（用户长短期记忆、个人知识图谱）
- 习惯层统计（轻量无模型，纯 SQL 聚合）
- 模型推荐（按硬件自动推荐可运行模型）
- 导出/导入（ZIP 包格式，含 SQLite + 文件夹结构）

## Demand Evidence

用户当前痛点：
1. 文献阅读工具无法提供知识整理
2. Markdown 笔记工具无法导入文献数据
3. 大模型工具（总结/回答）付费
4. 各工作流（文献、项目、个人知识）互相孤立，没有个人画像贯穿
5. 没有上下文的 AI 回复范围不受控，无法做精准的交叉验证

## Status Quo

用户目前使用分散工具链：
- 文献阅读：用某个能读 PDF 的工具（无 AI 整理）
- 笔记：用某个 Markdown 工具（无文献导入）
- AI 辅助：付费云端服务（无本地选项）
- 项目记忆：claude-mem 生成 .db/.wal（无法同步到本地知识库）

所有工作流孤立存在，AI 缺乏个人上下文，导致回答范围失控。

## Target User & Narrowest Wedge

**用户**：个人开发者/研究者，在 i5-13500H + 16GB RAM 笔记本上工作，需要管理文献、项目笔记、个人知识。

**最小可用版本（MVP）核心链路**：
1. 导入 Markdown 文件 → 存入板块文件夹 + SQLite 元数据
2. Chat 对话 → 注入板块内相关文件摘要作为上下文
3. 对话结束 → 自动生成 5000 字摘要存入 SQLite
4. 下次对话 → 自动引用板块历史摘要作为上下文

MVP 不含：画像系统、习惯统计、模型推荐。这些作为独立模块在后续迭代接入。

## Constraints

- **硬件**：i5-13500H + 16GB RAM，无独立 GPU
- **内存约束**：7B Q4_K 模型勉强运行（context 限制 2048），3B Q4_K 推荐，13B+ 不可行
- **模型路由**：用户主导的模型路由——用户设定默认路径（本地或云端），本地是隐私/离线选项，云端是性能选项
- **内存管理**：llama-server 优先加载（chat 延迟敏感），内存不足时减少 context 大小或切换至云端模型；FTS5 始终可用（内置），向量搜索（meilisearch）为可选增强层
- **模块化**：llama.cpp 作为独立模块集成，通过 Rust 接口调用，便于后续适配更新
- **数据隔离**：所有数据存在 App 私有目录，文件按板块组织在文件系统，AI 处理结果存 SQLite

## Premises

1. **知识分层存储**：文件本体（Markdown）存文件系统，AI 处理结果（摘要/标签/画像/chat 历史）存 SQLite，板块结构和文件索引也存 SQLite
2. **chat 摘要 vs 知识库摘要分开**：chat 层面是当前对话窗口的生命周期总结；知识库层面是文档导入后的结构化提取。两者最终都进入知识库，但触发时机和数据格式不同
3. **模型路由按任务类型**：习惯层统计（纯 SQL）、文件理解（本地或云端）、画像总结（云端优先）、chat 对话（用户自选）
4. **分层摘要解决上下文**：不通过扩大上下文窗口解决，而是通过注入板块相关摘要来分层解决
5. **板块专属 system prompt**：每个知识板块有独立模板，约束 AI 输出格式（如文献综述输出"标题/作者/核心发现"，项目复盘输出"技术决策/下一步"）

## llama.cpp 技术调研结论

### 内存占用对照表

| 量化格式 | 每 1B 参数 MB |
|---------|--------------|
| F16 | 2000 |
| Q8_0 | 1063 |
| Q6_K | 820 |
| Q5_K | 688 |
| Q4_K / Q4_0 | 563 |
| Q3_K | 430 |
| Q2_K | 328 |

### i5-13500H + 16GB RAM 可行性

| 模型 | 量化 | 内存 | Context | 可行性 |
|------|------|------|---------|--------|
| 1B | Q4_K | 563MB | 4096 | 流畅 |
| 3B | Q4_K | 1.7GB | 4096 | 推荐 |
| 7B | Q4_K | 3.9GB | 2048 | 勉强（需关闭其他程序） |
| 13B+ | - | - | - | 不可行 |

**推荐**：3B Q4_K 作为本地主力模型，7B Q4_K 作为可选（内存紧张）。

### llama.cpp 集成方案

**推荐：HTTP Server 模式（llama-server）**
- 独立 Python 进程，Rust 后端通过 HTTP 调用
- 与 llama.cpp 解耦，llama.cpp 升级不影响主应用
- API 清晰：`/completion`、`/chat/completions`、`/embedding`
- Session 通过 slot 机制管理

**备选：Direct FFI（未来性能优化）**
- 通过 `bindgen` 生成 Rust 绑定
- 直接调用 C 接口，无进程间开销
- 当前 MVP 选择 HTTP Server，后续可切换

### llama.cpp 模块化接口设计

**类型定义**（来自 `llama.cpp` crate）：
- `Token = i32`：token ID
- `Batch`：编码批次，含 token 数组、位置、序列信息
- `Vocab`：词汇表，通过 `ModelHandle` 获取
- `ProgressCallback = Fn(f32) -> bool`：模型加载进度回调，返 true 继续，false 中止

```rust
// 模型生命周期管理
trait ModelManager {
    fn load_model(path: &Path, gpu_layers: i32, cb: ProgressCallback) -> Result<ModelHandle, LlamaError>;
    fn unload_model(handle: ModelHandle);
    fn get_model_info(handle: ModelHandle) -> ModelInfo;
    fn list_downloadable_models() -> Vec<ModelListing>;  // from HuggingFace
}

// 推理调用
trait InferenceEngine {
    fn create_context(model: ModelHandle, ctx_size: u32, n_threads: u32) -> Result<ContextHandle, LlamaError>;
    fn encode(ctx: ContextHandle, tokens: &[Token]) -> Result<(), LlamaError>;
    fn decode(ctx: ContextHandle, batch: &Batch) -> Result<i32, LlamaError>;  // 返回值: 0成功, 1 KV满, 2中止, <0错误
    fn tokenize(vocab: &Vocab, text: &str, add_special: bool) -> Vec<Token>;
    fn detokenize(vocab: &Vocab, tokens: &[Token]) -> String;
}

// Session 状态管理
trait ContextPool {
    fn acquire_session(ctx: ContextHandle, session_id: SessionId) -> Session;
    fn release_session(session: Session);
    fn get_session_state(session: &Session) -> State;
    fn restore_session(session: &mut Session, state: State);
}

// 量化格式选择
trait QuantizationSelector {
    fn recommend_for_hardware(hw: &HardwareInfo) -> QuantizationConfig;
    fn list_available_quantizations() -> Vec<QuantizationInfo>;
    fn validate_compatibility(model_path: &Path, quant: &QuantizationConfig) -> bool;
}

// 模型下载
trait ModelDownloader {
    async fn download_model(model_id: &str, cb: ProgressCallback) -> Result<Path, LlamaError>;
    fn list_cached_models() -> Vec<CachedModel>;
    fn delete_model(model_id: &str);
}

// 硬件检测
trait HardwareDetector {
    fn detect_cpu() -> CpuInfo;
    fn detect_ram() -> RamInfo;
    fn detect_gpu() -> GpuInfo;
    fn estimate_capacity(hw: &HardwareInfo) -> ModelCapacity;
}
```

## Cross-Model Perspective

**独立 AI 视角（Claude subagent）评论：**

1. **Strongest version**：A personal research operating system that feels like a natural extension of your own cognition — where importing a PDF, asking questions, and building interconnected knowledge panels happens at the speed of thought, with local AI as the default and cloud as an invisible safety net.

2. **Most revealing insight**："Physics-unaware = seamless" 和 i5-13500H + 16GB RAM 约束共同揭示了真正的产品：让本地 AI 在受限硬件上感觉像云端一样快。

3. **Challenged premise**："本地优先"策略——在 16GB RAM 上 llama.cpp 推理可能比云端 API 慢，用户最终会转向云端，使"物理无感"变成营销话术而非实际体验。

**我的回应**：用户确认了云端作为主动选择路径，本地更多是隐私/离线能力。因此"本地优先"的定位调整为"用户主导的模型路由"——用户设定默认路径，本地和云端都是一等公民。

**结论**：第二视角的挑战被接受，但不影响推荐方案——"用户主导的模型路由"已经解决了这个问题。

## Approaches Considered

### Approach A: 最小核心链路（Fastest Path）

**核心模块**：
- 文件导入层（PDF/DOCX/MD/文件夹）
- 向量索引层（Chroma standalone，SQLite 元数据）
- Chat 界面（本地/云端模型切换）
- Chat 历史 + 摘要（SQLite，5000字基准）
- 板块管理（用户自定义板块 + 文件夹结构）
- 板块专属 system prompt
- 导出/导入（ZIP 包格式）

**Effort**: 中等（模块化，但核心链路工作量集中）
**Risk**: 低（核心链路清晰，无未知技术风险）

**Pros**: 4-6 周能看到实际成果；核心体验完整；所有架构决策在第一版就位
**Cons**: 画像系统、习惯统计、模型推荐不在第一版

### Approach B: 全功能同步交付（Full Scope）

所有功能并行开发： Approach A + 习惯层统计 + 画像层总结 + 模型推荐引擎 + claude-mem 数据同步。

**Effort**: 高（并行开发多个模块）
**Risk**: 中（习惯统计和画像系统的业务逻辑需要实际使用验证）

**Pros**: 第一版完整交付，无功能缺失
**Cons**: 开发周期长（10-16 周）；并行模块多，集成复杂度高

### Approach C: 分期交付（Phased Delivery）

Approach A 作为第一期（MVP），同时预留架构接口让画像/推荐系统可以在后续独立接入。第一期交付后立即可以使用，后续功能随用随加。

**Effort**: 低→中（分期投入）
**Risk**: 低（架构已为扩展预留接口）

**Pros**: 6-8 周交付第一期可用的核心；架构为扩展预留，迭代成本低；用户可以立即使用并提供反馈
**Cons**: 第一期功能比 B 少，但完整度足够日常使用

## Recommended Approach

**RECOMMENDATION: Approach C（分期交付）**

理由：
1. 需求已经很清晰，不需要"验证方向"——但需要"快速看到成果"来维持动力
2. "工业级标准流程"更多体现在架构设计上，而不是把所有功能一次性交付
3. 画像系统和模型推荐的实际效果需要真实使用数据来校准，第一期用真实数据跑起来后再调整，比空想更准确
4. 架构接口设计确保后续扩展不需要重写核心代码

### 决策记录

| 问题 | 决策 | 理由 |
|------|------|------|
| claude-mem 导入 | MVP 仅支持 Markdown | 降低第一版复杂度，先跑通核心链路再引入外部数据格式适配 |
| PDF 支持 | 文本提取优先 | pymupdf 处理扫描 PDF 需要额外 OCR 库（tesseract），增加依赖复杂度 |
| 向量检索 | FTS5 为主 + meilisearch 可选 | 避免 Python sidecar 打包风险（mem-switch 教训），FTS5 内置始终工作，meilisearch 作为增强层 |

## Success Criteria

1. **核心链路可跑通**：导入 Markdown → Chat 对话 → 自动摘要 → 再次对话引用历史摘要
2. **本地模型可用**：llama.cpp server 可在 16GB RAM 上跑 3B Q4_K 模型，响应时间 < 5s
3. **板块隔离有效**：不同板块的 chat 有独立上下文，不互相污染
4. **导出/导入完整**：ZIP 包可完整恢复所有数据和文件结构
5. **板块 prompt 有效**：同一模型在不同板块下输出格式不同（验证通过）

## Distribution Plan

### 数据模型

**SQLite 存储结构**：

| 表名 | 用途 |
|------|------|
| `panels` | 知识板块（id, name, system_prompt, created_at） |
| `files` | 文件索引（id, panel_id, path, title, created_at, vector_id） |
| `chat_sessions` | 对话 session（id, panel_id, created_at, updated_at） |
| `chat_messages` | 消息历史（id, session_id, role, content, created_at） |
| `summaries` | 自动摘要（id, panel_id, session_id, content, char_count, created_at） |
| `file_chunks` | 文件分块（id, file_id, chunk_index, content, embedding_ref） |

**文件系统结构**：
```
AppData/
├── files/           # 文件本体，按板块组织
│   ├── panel_1/
│   │   ├── doc1.md
│   │   └── doc2.md
│   └── panel_2/
│       └── note.md
├── meilisearch/     # meilisearch 数据目录（可选增强层）
└── knowledge.db     # SQLite 主数据库（含 FTS5）
```

**Success Criteria 量化定义**：
- **响应时间 < 5s**：warm query（模型已加载，无新文件检索），单轮对话
- **冷启动**：模型从磁盘加载到可响应的总时间
- **并发限制**：最多 3 个并发 chat session

**技术栈**：
- Tauri 2.x（桌面应用框架，沿用 knowledge-base 项目架构）
- Rust 后端（管理 llama-server 子进程，HTTP 调用）
- React 19 前端（Ant Design + TailwindCSS）
- SQLite + FTS5（结构化数据 + 内置全文搜索，始终可用）
- 文件系统（文件本体）
- meilisearch（向量搜索，可选增强层，单二进制）

**构建命令**：
```bash
pnpm install
pnpm tauri dev   # 开发模式
pnpm tauri build # 生产构建
```

**发布方式**：GitHub Releases，生成安装包（Windows/macOS/Linux）

**CI/CD**：GitHub Actions 自动构建和发布

## Dependencies

1. **llama.cpp**：模型推理核心，作为子进程（llama-server）管理
2. **meilisearch**：向量数据库，单二进制，作为可选增强层管理（启动失败不影响主功能）
3. **SQLite FTS5**：内置全文搜索，主搜索路径，始终可用
4. **HuggingFace**：模型下载来源
